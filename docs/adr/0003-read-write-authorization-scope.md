# ADR 0003: Gate protected write intent without adding writable overlay semantics

## Status

Accepted, partially superseded by [ADR 0007](0007-bypass-and-effect-based-authorization.md).

ADR 0007 supersedes this ADR's earlier assumption that sandboxfs remains read-only after protected write authorization. Treat this ADR's write-execution/read-only language as obsolete historical context; the current model is ADR 0007's bypass/effect model, where allowed write effects are forwarded to the backing filesystem and backing filesystem/kernel errno is authoritative.

## Context

`sandboxfs` is currently read-only for content writes and creation-style mutations such as `write`, `create`, `mkdir`, `unlink`, `rmdir`, and `rename`. The new per-file read/write authorization feature adds blocking user approval before protected reads or writes, but content-write persistence is a separate design problem.

## Decision

Read/write protection rules attach to **protected path scopes** in the sandbox namespace. A rule names a sandbox namespace path pattern; it is not keyed by underlying host path identity. A path with no glob metacharacters is exact. `<path>/*` protects the directory at `<path>` and direct children one component below it, but not deeper descendants. `<path>/**` protects the directory at `<path>` and its full recursive subtree. Protection path patterns are separate from grant path patterns, even though both use the same `*` within one component and `**` across components vocabulary. CLI control commands use separate read/write rule operations, such as `protect-read`, `protect-write`, `unprotect-read`, and `unprotect-write`. Combined read+write protection commands are not supported and should not be added later; callers must specify READ and WRITE protection separately so rule intent and audit logs stay explicit. Protection-rule configuration and listing are CLI-only. The Access TUI is only for pending authorization decisions and ergonomic pending-request actions: read/write allow/deny/do-nothing resolves the selected pending request without editing the protected path or grant path, while metadata edit-command executes a user-edited trusted metadata command and unblocks the original pending request. It does not manage protection settings. A protected read/write approval window must not offer path editing because changing the path would create broader authorization scope rather than simply resolving the observed pending request. Protection rules can be listed with `list-protection`, optionally filtered with `--read` and/or `--write`. The default with no filter is equivalent to `--read --write`; specifying both filters is valid and has the same behavior as the default. List output is deterministic/stably sorted by sandbox namespace path lexicographically, then by intent kind with READ before WRITE for the same path, while internal storage preserves creation order. Listing protection rules is a read-only control query: it is not a protection-rule outcome event, does not change state, and does not appear in the default audit log. The first read/write authorization slice also does not need a debug-mode log line for list queries; the CLI command response is the observable result. Adding a protection rule that already exists is an idempotent no-op for internal state: it returns successfully, creates no duplicate rule, and changes no internal rule state, but it still emits/logs an idempotent protection-rule event with `result=already-present`. Removing a protection rule that is not present is also an internal no-op that returns successfully and emits/logs an outcome event with `result=not-present`. Successful state-changing protection rule commands use `result=added` for protect and `result=removed` for unprotect. Protection rule events use structured grammar with verb `protect` or `unprotect`, explicit `kind=READ|WRITE`, `pattern=<sandbox-pattern>`, and one of the fixed result values; CLI command names remain split as `protect-read`, `protect-write`, `unprotect-read`, and `unprotect-write`, but event/log rendering does not encode the kind into the verb. Events/logs do not carry CLI-vs-TUI source fields for authorization control actions; sandboxfs should not add a CLI-vs-TUI source distinction later because that distinction has no authorization meaning. No-state-change control commands still emit/log their outcome because the event/log stream records what happened, not only state mutations. Unprotect commands delete only the exact same intent kind and pattern rule; removing `/secret/*` does not remove `/secret` or `/secret/**`. Internally, sandboxfs policy, authorization, IPC/state payloads, TUI, and request/operation log semantics operate on sandbox namespace paths plus attach IDs, not underlying host path subjects. Host paths exist only as explicit mapping input, private backend I/O resolution details, explicitly debug-only backend diagnostics, and a narrow attach lifecycle audit entry that maps `attach=<id>` to the user-supplied mountpoint for multi-attach debugging. Host paths are not authorization subjects and must not appear in pending requests, grants, decisions, ordinary operation events, Access TUI text, or user-visible authorization errors. Access TUI pending list rows and selected pending details include the request ID, attach ID when present, sandbox namespace path, and formatted filesystem operation description. Every sandbox event receives a compact numeric, session-local, non-reused event ID before best-effort distribution, regardless of whether that event is printed to a log. Numeric IDs are used instead of UUIDs because they are shorter in CLI/TUI/log output. Live-session non-reuse is the semantic guarantee; consumers must not depend on IDs starting at 1, being contiguous or monotonic, or being globally unique beyond the live session. Attach IDs and request/decision/cancel IDs follow the same live-session non-reuse rule. Logging is a consumer/rendering of selected events; it does not create separate identities. The general internal sandbox event stream is not limited by the pending-queue capacity policy. The only explicitly bounded queue in this slice is the pending authorization queue: pending metadata requests and pending read/write authorization requests awaiting user decision. This queue is session-wide, not per attach and not per pending kind; oldest-first eviction uses one global pending creation order across all attaches and pending types. Its default capacity is 100 and can be overridden with `SANDBOXFS_PENDING_EVENT_QUEUE_CAPACITY`; invalid values other than the explicit magic value `0` are a configuration error rather than silently falling back to the default. `0` disables the pending queue capacity limit and allows unlimited pending authorization requests; because this can leave many FUSE operations blocked, `sandboxfs run <name>` prints a startup warning when the effective capacity is unlimited. When a new metadata pending request arrives, sandboxfs first applies the metadata replacement rule for the same sandbox namespace path and pending operation kind, then checks session-wide pending queue capacity. When a new pending request would exceed a positive capacity after any metadata replacement, sandboxfs cancels the globally oldest pending request first using lifecycle cancel semantics, enqueues the new request, then re-checks capacity and repeats oldest-first cancellation if needed. Pending-capacity-limit cancellation is not user `DENY`; it returns `ECANCELED` to the canceled blocked operation and is logged/distributed as cancellation with the self-describing reason `pending-capacity-limit`. sandboxfs does not implement complex degraded-mode recovery for general event distribution failures in this slice: if event distribution or consumers fail, the failure is not hidden behind fallback state machinery. Because logging is handled by an independent loop/consumer, log writer failures affect log output only and must not downgrade, roll back, or otherwise change authorization/filesystem state. Default logs are audit/review oriented: they print protected attempts that become pending, user/system decisions, cancel cleanup, grant lifecycle events, downgrade warnings, and attach/detach/destroy lifecycle events. High-frequency filesystem trace events, including ordinary read/write operations that are already released by an active grant, are still distributed with event IDs but are printed only in `--debug` mode rather than the default audit log. Backend I/O failures should be rendered as sandbox namespace path + operation + errno in user-visible paths.

For the first read/write authorization slice, protected read attempts are **read intent authorizations** when they read protected regular-file contents or list a protected directory. `LOOKUP`/`GETATTR`-style existence and metadata probes are not read intent by default; gating those probes would create high-frequency blocking prompts and belongs behind a later, more explicit privacy/debug design if needed. Regular-file read authorization is evaluated at the actual file-content read operation, not at lookup/getattr/open-for-read, because opening a file handle does not expose bytes. Directory read authorization is evaluated when listing directory contents, such as `readdir`; opening a directory handle alone is not a protected read.

The authorization shape follows Linux fanotify permission events conceptually: the filesystem raises a pending read/write authorization request before the protected intent proceeds, and the operation waits for an explicit allow/deny decision rather than turning the protected attempt into an immediate permission failure. Multiple read/write requests may queue concurrently. Access TUI renders pending read/write requests as individual selectable requests with enough visible information for the user to see exactly what they are allowing or denying: request ID, attach ID when present, read/write intent kind, sandbox namespace path, formatted filesystem operation, and requester identity when available. The default TUI allow/deny/do-nothing action applies to the selected pending request only and keeps the existing key meanings `a=allow`, `d=deny`, and `n=do-nothing` for protected read/write requests. For protected read/write requests, Access TUI does not create future-matching grants and does not expose PID, process-tree, duration, batch, or path-scope grant controls; broader grants are deliberate CLI control actions. `e=edit` is not shown or enabled for protected read/write requests; edit-command is metadata-only because it executes a user-edited trusted metadata command and then releases the original metadata request. It must not hide a batch or broader grant behind a single ambiguous button. The TUI keeps do-nothing for protected read/write pending requests as an explicit escape hatch so the user can compose a manual CLI-side "magic" operation and then unblock the original request without creating a future grant or changing protection rules. The first read/write authorization slice does not provide a TUI batch action such as allowing all visible or all selected requests; batch authorization would need a later explicit selection UI, summary, confirmation text, and log semantics. A user allow with grant semantics broadcasts the allowed requester set, grant path pattern, and read/write grant intent kind to pending requests. The allowed requester set is the exact PID for requester PID grants, or the explicitly allowed process-tree PID set for process tree grants. Each pending request compares its requester, path, and intent kind against the broadcast; matching requests proceed, and nonmatching requests keep waiting. A deny decision applies only to the specified pending request: it does not broadcast, affect other waiting requests, or create a negative grant/blocklist. Authorization cancel is a lifecycle cleanup, not a user denial. It covers both pending metadata requests and pending read/write authorization requests, because both can leave FUSE operations blocked and prevent unmount. Sandbox destroy or session exit cancels all waiting authorization requests in that sandbox, unblocking their filesystem operations with `ECANCELED` and recording cleanup distinctly from a user `DENY`. User `DENY` keeps permission-denial behavior; lifecycle cleanup uses `ECANCELED` so callers and logs can distinguish cancellation from refusal. The explicit control syntax is `sandboxfs <name> cancel-all` for sandbox-wide cancellation and `sandboxfs <name> cancel-all <mountpoint>` for attached-view-scoped cancellation. Detach first performs the same attached-view-scoped cancellation for the mountpoint being detached, then unmounts: it cancels only requests that originated from that view, so a sandbox with multiple attached views does not cancel unrelated waiters from other attaches even when they involve the same sandbox namespace paths. The mountpoint parameter identifies the attached view for the control command. Internally, the attached view is represented by an attach ID; like other sandboxfs IDs, attach IDs are not reused within a live sandbox session. Requests, decisions, cancel events, filesystem-operation logs, and Access TUI pending request displays that originate from an attached view carry `attach=<id>` for multi-attach debugging. The attach lifecycle log entry may include the user-supplied mountpoint host path to map that host-visible mount to its attach ID, but later authorization and operation records use attach IDs and sandbox namespace paths rather than repeating host paths. Requests are also resolved by other request cleanup, such as closing the waiting operation, and by requester liveness events. Requester liveness is monitored by a dedicated session process/task/coroutine rather than by each blocked filesystem operation actively polling; blocked operations subscribe to authorization and liveness notifications, and a matching liveness event cancels only the affected pending request.

Allowing a read/write request may create a **read/write access grant** scoped to a grant path pattern, read/write grant intent kind, and requester subject. Grants live only in the live sandbox session's memory: they are not written to disk, are not restored across `sandboxfs run <name>` restarts, and are discarded on sandbox destroy or session exit. One-shot grants are removed immediately after consumption. Duration grants are removed from the active grant set when they expire. Reusable grants tied to a requester are removed when requester liveness/identity evidence says that requester is no longer alive. Default audit logs print low-frequency grant lifecycle events such as grant creation, one-shot consumption, duration expiry, requester-liveness pruning, and downgrade. High-frequency grant-hit or read/write pass-through events are distributed with event IDs but printed only in `--debug` mode. Reusable grants require requester identity evidence from a process-info provider that is stable enough to guard against PID reuse; sandboxfs should prefer a cross-platform provider/adapter such as `sysinfo` over hand-written Linux `/proc` parsing in core authorization logic. If stable requester identity evidence is unavailable, sandboxfs downgrades the allow to a grantless/current-request allow, reports a warning to the authorizing control plane, records the warning in the decision event, and must not silently create a reusable duration or future-matching grant from a raw PID alone.

The default requester subject is an exact requester PID, guarded by requester identity evidence; it does not include children, process groups, or login/session peers. A broader process-tree grant is available only through an explicit CLI choice, not through the default allow action and not through the first-slice Access TUI read/write approval UI. Access TUI read/write actions resolve only the selected pending request; they do not create PID, process-tree, duration, batch, or path-scope future grants. Process-tree grants use a snapshot of the requester process tree taken at allow decision time; future descendants are not automatically included and must trigger their own pending request or be covered by a later explicit allow. If the control plane requests a process-tree grant but the tree cannot be obtained, sandboxfs ignores the tree broadening, reports a warning to the authorizing control plane (CLI stderr), records the warning in the decision event, and falls back to the exact requester subject when reusable requester identity evidence is still available. The blocked FUSE caller should not receive this warning text; it is not the authorization operator.

Grant intent kinds are deliberately coarse: `READ` and `WRITE`. A `READ` grant covers protected file-content reads and protected directory listings. A `WRITE` grant covers protected data and namespace mutation attempts. `READ` grants do not cover `WRITE`, `WRITE` grants do not cover `READ`, and neither kind covers metadata authorization such as chmod, chown, or chattr.

Grant path patterns are user-created authorization scopes, separate from the protected path scopes that decide when a request becomes pending. A grant path pattern is interpreted against sandbox namespace paths only, with glob rules: a path with no glob metacharacters is exact, `*` matches within a path component, and `**` can match across path components. Recursive subtree grants are written as a glob such as `/secret/**`; there is no separate `--recursive` flag. CLI allow syntax puts the pending request ID immediately after `allow`, with optional path override after it, such as `sandboxfs <name> allow <id> --path /secret/**`. Access TUI does not expose grant path editing or recursive path normalization for protected read/write pending requests; any broader path grant is a deliberate CLI control action rather than an interactive TUI path edit.

For CLI use, sandboxfs should also support a **grantless read/write allow** that only releases the current blocked request without creating a read/write access grant; this is the read/write counterpart of an empty allow used to unblock the waiting operation. Access TUI exposes do-nothing for the selected protected read/write pending request as a deliberate escape hatch for users who first perform manual CLI-side remediation or another "magic" operation, then need to unblock the original request without creating a future grant or changing protection rules. Read/write do-nothing does not synthesize fake success, empty read data, or fake mutation results; after the waiter is released, the original operation continues through the same filesystem path as an allow. The difference from `ALLOW` is the audited/operator intent: `ALLOW` approves the observed read/write intent, while `ALLOW_DO_NOTHING` records that the user only unblocked the waiter without granting future access. Read/write do-nothing decision events use the existing `ALLOW_DO_NOTHING` decision kind; they are not user `DENY` decisions.

A **one-shot read/write grant** is consumed by the first matching protected filesystem intent in the pending queue or a later matching intent if no current pending request matches the broadcast. If a one-shot grant broadcast matches multiple pending requests at once, only the earliest-created matching request proceeds and the rest keep waiting. The filesystem does not know whether that intent belongs to `cat`, an editor, a compiler, or some other higher-level application command, so one-shot grants do not infer or group application-level operations. If a program performs multiple protected filesystem intents, it needs multiple one-shot grants unless a PID, process-tree, or duration grant covers them. Duration and other non-one-shot grant broadcasts release all currently matching pending requests.

A **duration read/write grant** starts its expiration clock when the user makes the allow decision, not when the pending request was created. `allow --duration` without an explicit duration uses the default 30 minutes. This prevents time spent waiting for the user to notice a pending request from reducing the effective authorized window. Duration does not make the grant global: it still uses the selected requester subject, defaulting to exact requester PID and expanding to process tree only when explicitly selected. If stable requester identity evidence is unavailable, a requested duration grant is downgraded to a grantless/current-request allow with a control-plane warning and decision-event warning. When a duration grant expires, it is removed from the active grant set and simply stops releasing future or still-waiting requests; expiry is not a deny broadcast and does not retroactively affect requests already released by the grant.

Protected write attempts are **write intent authorizations** for data and namespace mutation attempts: open-for-write, write, truncate, create, mkdir, unlink, rmdir, and rename. The filesystem blocks before a protected write-intent operation and waits for an allow/deny decision or matching grant broadcast, but an allow only permits the attempt to continue to the filesystem's existing behavior. Because sandboxfs remains read-only, that existing behavior may still return `EROFS` for writes or mutation attempts.

Metadata changes are deliberately not folded into write intent authorization. `chmod`, `chown`, `chattr`, `SETATTR`, and inode-flag ioctls remain in the existing Pending Metadata Request model with its own replacement and decision semantics. When a metadata pending request would replace an earlier same-path/same-kind request, replacement runs before session-wide pending queue capacity eviction so metadata domain de-duplication happens before unrelated oldest pending requests are canceled.

Writable overlay semantics are explicitly out of scope for this slice. Adding them later requires a separate decision for sandbox-local content storage, open writable handles, truncation, creation, deletion, rename, cleanup, and interaction with immutable metadata overrides.
