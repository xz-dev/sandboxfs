# Context Glossary

## Sandbox
A named, in-memory filesystem namespace. A sandbox disappears when its owning live session ends or it is destroyed.

## Sandbox Namespace Path
The canonical path identity used by sandboxfs policy, authorization, IPC/state payloads, TUI, and most logs. Internal authorization-facing data flow and audit records for requests, grants, decisions, and filesystem operations contain sandbox namespace paths, not underlying host paths. Host paths are confined to explicit mapping input, private backend I/O resolution details, debug-only diagnostics, and the attach lifecycle log entry that maps an attach ID to the user-supplied mountpoint for debugging. User-visible authorization errors should report the sandbox namespace path, operation, and errno rather than leaking backend host paths.

## Sandbox Session
The live lifecycle of one sandbox. A session begins when the user runs `sandboxfs run <name>` and ends when that foreground process exits or receives a destroy/shutdown control command.

## Sandbox Event
A structured notification emitted inside a live sandbox session before any log rendering. Every sandbox event receives a compact numeric, session-local, non-reused event ID before best-effort distribution, regardless of whether that event is printed to a log. The guarantee is live-session non-reuse, not UUID-style global uniqueness or contiguous/monotonic allocation. Logging is a consumer/rendering of selected events; events that are not written to the default audit log are still distributed with their event IDs, and `--debug` controls whether high-frequency trace events are printed. The general sandbox event stream is not capacity-limited by the pending-queue policy. sandboxfs does not implement complex degraded-mode recovery for general event distribution failures in the first slice: if event distribution or consumers fail, the failure is not hidden behind fallback state machinery. The explicit bounded-capacity fallback is limited to the pending authorization queue. Because logging is handled by an independent loop/consumer, log writer failures only affect log output and must not change authorization or filesystem state.

## Control Command
A short-lived user command that asks an existing foreground sandbox session to change or inspect the sandbox through that session's runtime socket.

## Attach
Exposing a sandbox at a host mountpoint so normal filesystem tools can access it.

## Attached Filesystem View
One live filesystem exposure created by an attach operation. A sandbox can have multiple attached filesystem views at the same time, so pending authorization cleanup for one view must not cancel requests that originated from another view. Control commands may identify a view by its mountpoint parameter, but authorization records remain tied to the attached view and sandbox namespace paths rather than treating host paths as authorization subjects.

## Attach ID
A session-local identifier assigned to an attached filesystem view. sandboxfs uses compact numeric IDs rather than UUIDs for user-visible handles. Like other sandboxfs IDs, attach IDs are not reused within a live sandbox session. Live-session non-reuse is the semantic guarantee; consumers must not depend on IDs starting at 1, being contiguous, or being globally unique beyond the live session. Requests, decisions, cancel events, filesystem operation logs, and Access TUI pending request displays that originate from an attached view carry `attach=<id>` so multi-attach debugging can distinguish otherwise identical sandbox namespace paths. The attach lifecycle log entry may include the user-supplied mountpoint host path to map that host-visible mount to its attach ID; later authorization and operation records refer to the attach ID instead of repeating the host path.

## Mapping
A rule that makes an existing local file or directory visible at a sandbox path.

## Hide Rule
A rule that removes a sandbox path and its descendants from visibility until a newer mapping covers that path again.

## Metadata Override
A sandbox-local metadata change, such as mode, owner, group, or flags, that does not change the underlying local file or directory.

## Pending Metadata Request
An untrusted metadata change requested through an attached filesystem that must be allowed or denied by the user. Viewing pending requests through the CLI or TUI is read-only and can happen concurrently; explicit allow/deny commands are what resolve pending state. Access TUI edit-command is not a command draft: it executes the user-edited trusted metadata command and releases the original blocked request with do-nothing so the original pending request unblocks. Authorization cancel can also release pending metadata waiters as lifecycle cleanup rather than as a user denial.

## Protected Path Scope
A sandbox namespace path pattern that decides whether an operation becomes a protected read or write intent. Protection is evaluated only against sandbox namespace paths, never by underlying host path identity. A plain sandbox path with no glob metacharacters is exact. `<path>/*` protects the directory at `<path>` and direct children one component below it, but not deeper descendants. `<path>/**` protects the directory at `<path>` and its full recursive subtree. Protection path patterns are separate from grant path patterns, even though both use the same `*` within one component and `**` across components vocabulary. CLI control commands use separate read/write rule operations, such as `protect-read`, `protect-write`, `unprotect-read`, and `unprotect-write`. Combined read+write protection commands are not supported and should not be added later; callers must specify READ and WRITE protection separately so rule intent and audit logs stay explicit. Protection-rule configuration and listing are CLI-only. The Access TUI is only for pending authorization decisions and ergonomic pending-request actions: read/write allow/deny/do-nothing resolves the selected pending request without editing the protected path or grant path, while metadata edit-command executes a user-edited trusted metadata command and unblocks the original pending request. It does not manage protection settings. A protected read/write approval window must not offer path editing because changing the path would create broader authorization scope rather than simply resolving the observed pending request. Protection rules can be listed with `list-protection`, optionally filtered with `--read` and/or `--write`. The default with no filter is equivalent to `--read --write`; specifying both filters is valid and has the same behavior as the default. List output is deterministic/stably sorted by sandbox namespace path lexicographically, then by intent kind with READ before WRITE for the same path, while internal storage preserves creation order. Listing protection rules is a read-only control query: it is not a protection-rule outcome event, does not change state, and does not appear in the default audit log. The first read/write authorization slice also does not need a debug-mode log line for list queries; the CLI command response is the observable result. Adding a protection rule that already exists is an idempotent no-op for internal state: it returns successfully, creates no duplicate rule, and changes no internal rule state, but it still emits/logs an idempotent protection-rule event with `result=already-present`. Removing a protection rule that is not present is also an internal no-op that returns successfully and emits/logs an outcome event with `result=not-present`. Successful state-changing protection rule commands use `result=added` for protect and `result=removed` for unprotect. Protection rule events use structured grammar with verb `protect` or `unprotect`, explicit `kind=READ|WRITE`, `pattern=<sandbox-pattern>`, and one of the fixed result values; CLI command names remain split as `protect-read`, `protect-write`, `unprotect-read`, and `unprotect-write`, but event/log rendering does not encode the kind into the verb. Events/logs do not carry CLI-vs-TUI source fields for authorization control actions; sandboxfs should not add a CLI-vs-TUI source distinction later because that distinction has no authorization meaning. No-state-change control commands still emit/log their outcome because the event/log stream records what happened, not only state mutations. Unprotect commands delete only the exact same intent kind and pattern rule; removing `/secret/*` does not remove `/secret` or `/secret/**`.

## Read Intent Authorization
A pending decision for an untrusted attempt to read protected file contents or list a protected directory. Existence and metadata probes are not read intent by default. Regular-file read authorization is evaluated at the actual file-content read operation, not at lookup/getattr/open-for-read, because opening a file handle does not expose bytes. Directory read authorization is evaluated when listing directory contents, such as `readdir`; opening a directory handle alone is not a protected read.

## Write Intent Authorization
A pending decision for an untrusted attempt to perform a protected data or namespace mutation, such as open-for-write, write, truncate, create, mkdir, unlink, rmdir, or rename. Allowing the attempt lets it continue to the filesystem's existing write behavior; it is not a writable overlay permission, does not make sandbox content mutable, and does not replace the metadata authorization model.

## Pending Read/Write Authorization Request
A fanotify-like pending request raised before a protected read or write intent proceeds. Multiple read/write requests may queue concurrently. Access TUI renders pending read/write requests as individual selectable requests with enough visible information for the user to see exactly what they are allowing or denying: request ID, attach ID when present, read/write intent kind, sandbox namespace path, formatted filesystem operation, and requester identity when available. The default TUI allow/deny/do-nothing action applies to the selected pending request only and keeps the existing key meanings `a=allow`, `d=deny`, and `n=do-nothing` for protected read/write requests. For protected read/write requests, Access TUI does not create future-matching grants and does not expose PID, process-tree, duration, batch, or path-scope grant controls; broader grants are deliberate CLI control actions. `e=edit` is not shown or enabled for protected read/write requests; edit-command is metadata-only because it executes a user-edited trusted metadata command and then releases the original metadata request. It must not hide a batch or broader grant behind a single ambiguous button. The TUI keeps do-nothing for protected read/write pending requests as an explicit escape hatch so the user can compose a manual CLI-side "magic" operation and then unblock the original request without creating a future grant or changing protection rules. The first read/write authorization slice does not provide a TUI batch action such as allowing all visible or all selected requests; batch authorization would need a later explicit selection UI, summary, confirmation text, and log semantics. The filesystem operation waits for an explicit allow/deny decision or a matching read/write grant broadcast instead of being auto-denied like an ordinary permission failure; session or request closure cleans up the wait. When a grant broadcast arrives, each pending request compares its requester, path, and intent kind against the broadcast; matching requests proceed and nonmatching requests keep waiting. A deny decision applies only to the specified pending request; it does not broadcast, affect other waiting requests, or create a negative grant/blocklist.

## Pending Event Queue
The bounded in-memory queue of pending authorization events/requests awaiting user decision. This is the only queue limited by `SANDBOXFS_PENDING_EVENT_QUEUE_CAPACITY`; the general internal sandbox event stream is not limited by this setting. The pending queue covers pending metadata requests and pending read/write authorization requests across the whole sandbox session. Capacity is session-wide, not per attach and not per pending kind; oldest-first eviction uses one global pending creation order across all attaches and pending types. The default capacity is 100 and can be overridden with `SANDBOXFS_PENDING_EVENT_QUEUE_CAPACITY`; invalid values other than the explicit magic value `0` are a configuration error rather than silently falling back to the default, so users do not mistakenly believe their requested limit was applied. `0` disables the pending queue capacity limit and allows unlimited pending authorization requests; because this can leave many FUSE operations blocked, `sandboxfs run <name>` prints a startup warning when the effective capacity is unlimited. When a new metadata pending request arrives, sandboxfs first applies the metadata replacement rule for the same sandbox namespace path and pending operation kind, then checks session-wide pending queue capacity. When a new pending request would exceed a positive capacity after any metadata replacement, sandboxfs cancels the globally oldest pending request first using lifecycle cancel semantics, enqueues the new request, then re-checks capacity and repeats oldest-first cancellation if needed. Pending-capacity-limit cancellation is not user `DENY`; it returns `ECANCELED` to the canceled blocked operation and should be logged/distributed as cancellation with the self-describing reason `pending-capacity-limit`.

## Authorization Cancel
A lifecycle cleanup that releases pending authorization waiters without treating the result as a user denial. It covers both pending metadata requests and pending read/write authorization requests, because both can leave FUSE operations blocked. Canceled blocked filesystem operations return `ECANCELED`, distinguishing lifecycle cleanup from user `DENY` permission-denial behavior. Sandbox destroy or session exit cancels all waiting authorization requests in that sandbox. The explicit control syntax is `sandboxfs <name> cancel-all` for sandbox-wide cancellation and `sandboxfs <name> cancel-all <mountpoint>` for attached-view-scoped cancellation. Detach first performs attached-view-scoped cancellation for the mountpoint being detached, then unmounts: it cancels only requests that originated from that view, while requests from other attached views keep waiting even if they involve the same sandbox namespace paths. Pending event queue capacity eviction also uses authorization cancel semantics, canceling oldest pending requests with the self-describing reason `pending-capacity-limit`. Cancel logs include the attach ID when cancellation is attached-view-scoped, such as `cancel attach=<id> scope=attached-view reason=detach`; each canceled request also gets a separate request-level cancel event such as `cancel request=<request-id> attach=<id>`.

## Read/Write Access Grant
A temporary authorization created from allowing a protected read or write request. It is scoped to a grant path pattern, read/write grant intent kind, and requester subject rather than being a global sandbox permission. Grants live only in the live sandbox session's memory; they are not written to disk, are not restored across `sandboxfs run <name>` restarts, and are discarded on sandbox destroy or session exit. Reusable grants require requester identity evidence that is stable enough to guard against PID reuse; without that evidence, sandboxfs downgrades the allow to release only the current pending request and reports a control-plane warning. Reusable grants tied to a requester are removed when requester liveness/identity evidence says that requester is no longer alive. Default audit logs record low-frequency grant lifecycle events such as grant creation, one-shot consumption, duration expiry, requester-liveness pruning, and downgrade; high-frequency grant-hit/read-write pass-through events are `--debug`-only.

## Read/Write Grant Intent Kind
The operation class matched by a read/write access grant. There are two grant intent kinds: `READ`, covering protected file-content reads and protected directory listings; and `WRITE`, covering protected data and namespace mutation attempts. `READ` grants do not cover `WRITE`, `WRITE` grants do not cover `READ`, and neither grant kind covers metadata authorization such as chmod, chown, or chattr.

## Read/Write Grant Broadcast
The in-session notification emitted after a user allows a read/write authorization request with grant semantics. It carries the allowed requester set, grant path pattern, and read/write grant intent kind. For requester PID grants, the requester set is the allowed PID. For process tree grants, it is the explicitly allowed process-tree PID set. Pending read/write authorization requests use this broadcast to decide whether they can proceed or must continue waiting. A one-shot grant broadcast releases only the earliest matching pending request; duration and other non-one-shot grant broadcasts release all currently matching pending requests.

## Requester Liveness Event
An in-session notification that a requester process identity is no longer alive. It is produced by a dedicated liveness monitor rather than by each blocked filesystem operation actively polling. Pending read/write authorization requests subscribe to authorization and liveness notifications; a matching requester liveness event cancels only that pending request, while unrelated pending requests continue waiting. Reusable read/write grants tied to that requester are also pruned when the liveness event is processed.

## Grant Path Pattern
The sandbox namespace path glob pattern used by a read/write access grant. A plain sandbox path with no glob metacharacters is an exact match. `*` and `**` follow glob rules: `*` matches within a path component, while `**` can match across path components; `/secret/**` is the recursive subtree form. Matching is always against sandbox namespace paths, never host paths. CLI allow syntax puts the pending request ID immediately after `allow`, with the pattern supplied by `--path`, such as `sandboxfs <name> allow <id> --path /secret/**`; there is no separate `--recursive` flag. Access TUI does not expose grant path editing or recursive path normalization for protected read/write pending requests; any broader path grant is a deliberate CLI control action rather than an interactive TUI path edit.

## Grantless Read/Write Allow
An allow/do-nothing decision that releases the current blocked read/write authorization request without creating a read/write access grant. This is the read/write counterpart of an empty allow used only to unblock the waiting filesystem operation. Access TUI exposes do-nothing for the selected protected read/write pending request as a deliberate escape hatch for users who first perform manual CLI-side remediation or another "magic" operation, then need to unblock the original request without creating a future grant. Read/write do-nothing does not synthesize fake success, empty read data, or fake mutation results; after the waiter is released, the original operation continues through the same filesystem path as an allow. The difference from `ALLOW` is the audited/operator intent: `ALLOW` approves the observed read/write intent, while `ALLOW_DO_NOTHING` records that the user only unblocked the waiter without granting future access. Read/write do-nothing decision events use the existing `ALLOW_DO_NOTHING` decision kind. It is also the safe downgrade when a requested reusable grant lacks stable requester identity evidence.

## Requester Identity Evidence
Stable process identity facts supplied by a process-info provider for matching read/write grants across filesystem requests. It must be stronger than a raw PID when a grant can be reused, because raw PIDs can be recycled.

## Requester PID Grant
A read/write access grant whose requester subject is only the exact process that raised the pending request, guarded by requester identity evidence. It does not include children, process groups, or login/session peers.

## Process Tree Grant
A read/write access grant whose requester subject intentionally includes the requesting process's process tree. It is broader than a requester PID grant and must come from an explicit CLI choice rather than the default allow action. Access TUI does not expose process-tree grants in the first read/write authorization slice because TUI read/write actions resolve only the selected pending request. The process tree is a snapshot taken at allow decision time; future descendants are not automatically included. If the control plane requests a process-tree grant but the tree cannot be obtained, sandboxfs ignores the tree broadening, reports a control-plane warning, and falls back to the exact requester subject when reusable requester identity evidence is still available.

## One-Shot Read/Write Grant
A read/write access grant consumed by the next single matching protected filesystem intent visible to sandboxfs, then immediately removed from the active grant set. If a one-shot grant broadcast matches multiple pending requests, only the earliest-created matching pending request proceeds; the rest keep waiting. It does not infer or group higher-level application commands; a program that performs multiple protected filesystem intents needs multiple one-shot grants unless another grant covers them.

## Duration Read/Write Grant
A read/write access grant with an expiration time. Its lifetime starts when the user makes the allow decision, not when the pending request was created. If the user omits a duration, the default duration is 30 minutes. It still has a requester subject: exact requester PID by default, or process tree only when explicitly selected. If stable requester identity evidence is unavailable, sandboxfs downgrades a requested duration grant to a grantless allow and reports a control-plane warning. Expiration removes the grant from the active grant set and only stops it from releasing future or still-waiting requests; it is not a deny broadcast.

## Trusted Metadata Operation
A metadata operation initiated through sandboxfs itself, which skips the pending authorization flow but can still fail normally. Access TUI edit-command runs the user-edited `chmod`, `chown`, or `chattr` as a trusted metadata operation through the sandboxfs CLI path, then releases the original pending metadata request with do-nothing rather than leaving the original FUSE operation blocked.

## Operation Event
A filesystem-operation-specific sandbox event used before string formatting. Operation events carry an event ID, timestamp, sandbox, target sandbox namespace path, operation kind, operation fields, and attach ID when the operation originates from an attached filesystem view. Logs and TUI text are renderings of operation events, not reconstructions of shell commands. Access TUI pending list rows and selected pending details include the attach ID when present, alongside the request ID, sandbox namespace path, and formatted filesystem operation description. Operation events must not carry host paths except for the attach lifecycle event that maps an attach ID to its mountpoint.

## Audit Log Entry
A serialized rendering of a sandbox event written by the session log writer. Every audit log entry has a bracketed UTC timestamp with microsecond precision and prints the event's `id=<event-id>`; the log does not create a separate identity for the event. Authorization and filesystem-operation log entries render sandbox namespace paths plus attach IDs when applicable. Default logs are audit/review oriented: they record protected attempts that become pending, user/system decisions, cancel cleanup, grant lifecycle events, downgrade warnings, and attach/detach/destroy lifecycle events. High-frequency filesystem trace events, including ordinary read/write operations that are already released by an active grant, are still distributed as events with IDs but are printed only in `--debug` mode rather than the default audit log. Host paths are excluded from ordinary logs, including backend I/O failure messages; the narrow exception is the attach lifecycle log entry that maps `attach=<id>` to the user-supplied mountpoint for multi-attach debugging.

## Debug Backend Diagnostic
A diagnostic or trace sandbox event printed only when `--debug` logging is enabled. It may include high-frequency filesystem details and private backend details such as host paths for troubleshooting path resolution or I/O bugs. Debug mode is where ordinary grant-allowed read/write pass-through events and backend resolution/I/O diagnostics are printed. These events still receive event IDs and are distributed consistently even when they are not printed to the default audit log.

## Log Writer
The single writer loop responsible for resetting, appending to, and removing sandbox log files. Filesystem handlers and control handlers publish log events to the writer instead of writing log files directly. The log writer is an event consumer; log output is best-effort rendering of sandbox events rather than the source of event identity. If the log writer, log queue, or log file write fails, that failure affects logging only; it does not downgrade, roll back, or otherwise change authorization/filesystem state. The first slice intentionally avoids complex degraded-mode recovery or fallback logging paths if the writer/queue fails.

## Decision Event
A log event recording a user or system authorization decision. It has its own event ID and references the pending request with `request=<pending-id>`. Decision kinds are `ALLOW`, `ALLOW_DO_NOTHING`, and `DENY`. For read/write authorization, TUI/CLI do-nothing uses `ALLOW_DO_NOTHING`; it is not user `DENY` and does not create a future grant. For read/write authorization, `DENY` is scoped to the referenced pending request only and is not a negative grant. If an allow request is downgraded, such as a process-tree grant falling back to an exact requester grant, the decision event records the warning. Authorization cancel events are recorded distinctly from user `DENY` decisions.

## Pending Operation Kind
The replacement dimension for pending metadata authorization on a target path. Current kinds are `mode`, `uid`, `gid`, and `flags`. A later request with the same sandbox, path, and kind denies and replaces the earlier pending request before session-wide pending queue capacity is checked; different kinds can remain pending independently.

## Superseded Pending Request
A pending request that was automatically denied because a later request with the same sandbox, target path, and operation kind arrived. This is behaviorally the same as the user denying the earlier request before the later request becomes pending.

## Ext4-like Open File Semantics
Metadata changes should match ext4 behavior as closely as practical: chmod/chown do not revoke already-open file descriptors, new opens observe allowed metadata overrides, and immutable flags should reject later writes once applied.
